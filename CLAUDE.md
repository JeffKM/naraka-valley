# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 이 저장소의 성격 (먼저 읽을 것)

**나라카 밸리**는 저승 컨셉카페 세계관의 **진짜 스타듀밸리형 게임**(농사·낚시·채광·관계)이다. 엔진은 **Godot**, 타겟은 **Steam/PC**.

현재 저장소는 **기획 단계**다. 아직 Godot 프로젝트(게임 코드)가 생성되지 않았고, 저장소에는 기획 문서·결정 기록·작업 도구만 있다. 첫 코드는 ROADMAP의 **Phase 1 / Sprint 1**(Godot 그레이박스)에서 시작된다.

이 저장소는 **Git으로 버전 관리된다.** 원격은 `github.com/JeffKM/naraka-valley`(기본 브랜치 `main`)다. Git 작업은 `.claude/commands/`의 `/git:branch` `/git:commit` `/git:merge` `/git:pr` 워크플로우를 따른다.

> **★ 구현 착수 = 워크트리 격리부터(어길 수 없는 규칙).** owner는 여러 Claude 세션을 **같은 워크트리에서 동시에** 돌린다. 공유 워크트리에서 각 세션이 브랜치를 전환·편집하면 서로의 변경이 사라져 보이고(실제론 각 브랜치에 온전), Godot `user://save.dat`(앱명 공유)까지 경합해 세이브/복원 테스트가 코드와 무관하게 오탐 실패한다. **그러므로 코드·에셋을 수정하는 구현/빌드 작업을 시작하기 전에, 맨 먼저 `EnterWorktree`로 격리 워크트리를 만들고 그 안에서 작업한다**(이 규칙이 곧 `EnterWorktree` 사용 인가다 — 도구 설명이 "CLAUDE.md/메모리가 지시하면 사용"이라 명시). 순서: ①`EnterWorktree`(격리 워크트리 생성·진입) → ②새 워크트리는 `.godot`가 비어 있으니 첫 헤드리스 실행 전 `godot --headless --import` 1회 → ③구현·`game/run_tests.sh` 회귀 → ④`/git:commit`·`/git:pr` → ⑤`ExitWorktree`(keep=나중 이어감 / remove=완료). **예외(워크트리 불요):** ROADMAP·문서·메모리만 고치는 소규모 편집, 또는 이미 격리 워크트리 안일 때. **금지:** 같은 테스트를 두 워크트리에서 동시 헤드리스 실행(save.dat 경합 — 순차로).

> **벤더 디렉터리 주의:** `mcp-shrimp-task-manager/`는 upstream(cjo4m06)을 일반 파일로 벤더링한 것이다. 서브모듈이 아니므로 내부에 `.git`을 두지 말 것. `node_modules/`·`dist/`·`shrimp_data/`는 `.gitignore`로 제외되어 있어, 클론 후 도구를 쓰려면 `cd mcp-shrimp-task-manager && npm install && npm run build`가 필요하다.

## 어겨선 안 되는 설계 제약 (ADR 요약)

- **ADR-0002 — 엔진은 Godot/GDScript.** 게임 본체를 웹 스택(Phaser 등)으로 만들지 않는다. 개발자는 TS/React 전문가지만 게임은 Godot으로 학습하며 만든다.
- **ADR-0003(→ ADR-0012로 2배 개정) — 픽셀 규격:** 탑다운 3/4뷰·치비 ~2.5등신·정수배 스케일·표정=**대화 시 별도 일러스트 초상화**는 유효. **수치 규격은 ADR-0012가 2배 상향**: 타일 아트 16×16(렌더 ×2=32px) / 캐릭터 ~32×64(시트 프레임 80px) / 내부해상도 **640×360**. 인게임 얼굴 가독성(플레이테스트 1순위) 때문. **캐릭터 생성=PixelLab `mode=standard, size=56, detail=high` + 공통 `proportions` 고정값**(체형 통일 — v3는 비율 제각각이라 폐기). 새 환경 아트=16px 원본(렌더 시 ×2).
- **ADR-0004 — "속죄" 테마가 게임을 관통한다.** 각 캐릭터는 생전의 죄의 힘을 정반대로 써서 속죄한다(미호: 방화→작물 양육, 멜: 도박·사채→카페 운영(ADR-0007), 바나: 주거침입·흡혈→밤 경비). **이 테마를 어기는 캐릭터/시스템은 추가하지 않는다.**
- **ADR-0001 — 자체 "도트화 툴"을 만들지 않는다.** 인게임 도트는 PixelLab/Retro Diffusion으로 생성하고 Aseprite로 보정한다. 에셋 정리·임포트용 글루 스크립트는 허용하되, 변환 엔진 제작(야크 셰이빙)은 금지.
- **ADR-0005 — 메인 스토리는 (A) 창발형.** 강제 줄거리 게임(B)이 아니라 스타듀식으로, 이야기 = 캐릭터 속죄 서사들의 총합 + 플레이어의 미결의 죄. 플레이어는 기억상실이 아니며 죄=옥자와의 생전 인연(옥자가 앵커). **메인 서사 텍스트는 캐릭터에만 붙이고, 새 활동 시스템은 메인 플롯 글쓰기에 의존하지 않고 독립 완성한다.**
- **ADR-0008 — 성장 모델은 "base 메카닉 + 관계 곱셈기, 이야기는 게이트".** 관계(여우불류)는 *게이트가 아니라 곱셈기*다 — 모든 활동 메카닉은 ♡0에서도 base rate로 굴러가고("평평 ≠ 막힘", 은둔 활동파도 안 막힘), 관계는 그 위에 *항상 명백히 우월한* 가속을 얹는다(관계 = 의도된 최적 경로). **이야기(속죄 서사)만 캐릭터 호감도로 게이팅한다.** 곱셈기는 캐릭터마다 *종류*가 다르다(미호=자동화·멜=마진·바나=보호 — 같은 +%의 반복은 그라인드). 관계를 메카닉 게이트로 만들거나 순수 스타듀(관계=순수 덤)로 되돌리지 않는다.

## 개발 원칙

> ⚠️ **2026-06-29 개정([ADR-0028](./docs/adr/0028-build-workflow-greybox-art-interleave-no-fun-gate.md)):** 게임은 *무조건 출시*하므로 **재미 게이트(go/no-go 검증) 폐지**, 옥자 데모 일정 대폭 연기. 빌드 워크플로우가 아래로 바뀐다.

- **빌드 워크플로우 = 시스템별 "그레이박스 메카닉 → 아트" 인터리브** (ADR-0028). 메카닉 시스템은 *설계(그릴)→그레이박스로 로직·에셋 스펙 잠금→스타듀 아트→다음 시스템*. "전부 그레이박스 후 전부 아트"도 "아트 먼저"도 아니다. **순수 비주얼**(월드 룩·초상화·가구/장식)은 아트-주도.
- **그레이박스는 *검증 게이트가 아니라 빌드 스캐폴드*** — 아트 막힘 없이 로직을 짜고 에셋 스펙을 잠그는 수단(ADR-0025 스펙 카드의 입력). 한 시스템 안에서 그레이박스→아트로 이어진다.
- **현재 진행(2026-06-29~):** 스타듀 위키 전수 그릴로 *전 시스템 설계 확정* → **ROADMAP 전면 재배치·확장** → 빌드 착수. 설계가 굳기 전엔 빌드/아트에 안 들어간다(재작업 회피). 결정 누적 = `docs/design/stardew-systems-catalog.md`.
- **한 시스템을 100%로 끝내고 다음으로.** "전부 동시에"가 아니라 하나씩.
- 막힐 때 우선순위: **"끝까지 플레이되는 것" > "기능이 많은 것" > "예쁜 것".**

## 웹 코드와 게임 코드의 분리 (중요)

`.claude/rules/`의 규칙들(`api-routes.md`, `supabase.md`, `ui-components.md`)은 Next.js/Supabase 기준이며 **게임 본체가 아니라 홍보 홈페이지·위시리스트·웹 데모 쪽에만 적용된다**(ADR-0002 참조). Godot/GDScript 게임 코드에는 이 레이어드 아키텍처·Supabase·shadcn 규칙을 적용하지 말 것. 두 영역을 혼동하지 않는다.
