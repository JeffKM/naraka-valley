# P2.0 스파이크 — PixelLab 프롬프트 초안

> 재시작·MCP 승인 후 그대로 호출/복붙할 수 있는 초안. 프롬프트 문자열은 **영어**(PixelLab 모델이 영어 학습이라 결과가 나음), 설명은 한국어.
> 목표 = 얇은 단면(미호 + 혼령초 3단계 + 풀/길/흙 타일)으로 PASS 4기준 판정 + 스타일 가이드 잠금.
> 산출 PNG는 `game/spike/assets/`에 README의 이름대로 저장 → `asset_preview.tscn`(F6)로 적용·판정.

## 0. 스타일 잠금(모든 프롬프트에 공통으로 녹일 것)

- **규격:** 캐릭터 16×32(약 2.5등신 치비), 타일 16×16, 탑다운 3/4뷰, 내부해상도 320×180(정수배). (ADR-0003)
- **외곽선 = 역할별 선택적:** 전경(캐릭터·작물) = *어두운 같은 계열색* 외곽선(순검정 아님) / 배경(타일) = **외곽선 없음**(텍스처+음영, 격자 떡짐 방지).
- **팔레트 의도:** 저승·명부 톤 — 차분한 채도 + 푸른 "영혼빛" 액센트. 색 수 절제(limited palette). *정확한 마스터 팔레트는 스파이크가 발견해 잠근다.*
- **음영:** 2~3단 램프(과한 그라데이션 금지).

> 아래 프롬프트엔 위 의도를 영어로 박아뒀다. 1차 결과를 보고 팔레트/외곽선 강도를 조여 재호출 → 통하는 레시피를 이 문서에 확정 기록한다.

## 1. 미호 — 4방향 워크 (MCP: `create_character` → `animate_character`)

미호 정체성(CONTEXT): 구미호, 노란 한복, 여우귀, 아홉 꼬리·여우불(파란 불꽃), 밝고 해맑음.

```
create_character(
  description="cute cheerful nine-tailed fox girl, yellow hanbok (Korean dress),
    fox ears, hint of fluffy fox tails, small blue foxfire spark accent,
    top-down 3/4 JRPG view, ~2.5 head chibi proportions, 16x32 pixel sprite,
    selective dark tonal outline (not pure black) on silhouette, limited muted
    underworld palette with soft blue spirit accent, 2-3 tone shading, clean readable at small size",
  n_directions=4
)
# → 반환된 character_id로:
animate_character(character_id="<위 id>", animation="walk")
```

- 저장: 4방향 워크 시트를 `miho_walk.png`로. 시트 행/열 배치가 하네스 가정(행=방향 down/up/right/left, 열=프레임)과 다르면 `asset_preview.gd`의 `CHAR_DIRS`·`CHAR_FRAMES_PER_DIR` 조정.
- 판정 포인트: ③ 정체성(여우귀·한복이 16×32에서 읽히나) + ① 4프레임 일관성.

## 2. 혼령초 3단계 (REST API 일반 생성 — PixFlux/BitForge text-to-pixel)

MCP 4툴엔 임의 오브젝트 생성이 없으니 **API 일반 생성 엔드포인트**로(작물·소품은 전부 이 경로). 혼령초(CONTEXT): "푸르게 빛나는 영혼 풀", 빠름·저수익 입문 작물.

```
# 단계별 1장씩, 16x16
seed   : "tiny planted seed mound, top-down, 16x16 pixel, dark tonal outline,
          muted soil with faint blue spirit glow, 2-tone shading"
sprout : "small glowing blue spirit grass sprout, two thin luminous blades,
          top-down, 16x16 pixel, dark tonal outline, soft blue glow, 2-3 tone shading"
mature : "harvest-ready glowing blue spirit grass, full luminous tuft, clearly distinct
          from sprout, top-down, 16x16 pixel, dark tonal outline, bright blue spirit glow"
```

- 저장: `honryeong_seed.png` / `honryeong_sprout.png` / `honryeong_mature.png`.
- 판정 포인트: 세 단계가 *한눈에 구분*되나(특히 수확가능). 전경 외곽선이 타일 위에서 사나.

## 3. 풀↔길↔흙 타일 (MCP: `create_tileset` — Wang 전환)

P2.3에서 정한 "중간 오토타일(경계·모서리 전환)"의 스파이크판. 배경이므로 **외곽선 없음**.

```
create_tileset(lower="dirt path", upper="grass")     # 풀↔흙길 전환
create_tileset(lower="tilled farm soil", upper="dirt path")  # 길↔밭흙 전환
# 공통 의도: top-down 16x16, no black per-tile outline, textured fill with 2-3 tone shading,
#            muted underworld palette, seamless tiling
```

- 저장: 전환 세트에서 베이스 타일 3종을 잘라 `tile_ground.png`(풀) / `tile_path.png`(흙길) / `tile_soil.png`(밭흙). (전환 모서리 타일은 P2.3 본작업에서 Godot terrain set으로.)
- 판정 포인트: ② 타일 위에 미호·혼령초가 묻히지 않고 붙나 + 타일 이음매·격자 떡짐 없나.

## 4. 판정 결과 — ✅ PASS (2026-06-21, 잠금)

미호 + 혼령초 3단계 + 풀/흙길/밭흙 타일 7에셋을 `asset_preview.tscn`에서 PASS 4기준 통과(가독성·톤결합·미호 정체성·비용 체감). 아래 레시피를 **P2.1~P2.3 벌크가 그대로 복제**한다.

### 4.1 통한 프롬프트·파라미터 최종본 (실제 호출값)

**미호 — `create_character` → `animate_character`** (PixelLab MCP)
```
create_character(
  description="cute cheerful nine-tailed fox girl, bright yellow hanbok (Korean traditional dress),
    fox ears, fluffy fox tails, small blue foxfire spark accent, muted underworld palette with
    soft blue spirit accent, selective dark tonal outline not pure black, 2-3 tone shading,
    clean and readable at small size",
  n_directions=4, size=32, view="low top-down",
  outline="selective outline", shading="basic shading", detail="high detail",
  proportions={"type":"preset","name":"chibi"}
)
animate_character(character_id="<id>", template_animation_id="walk")   # 4방향, template 모드
```

**혼령초 3단계 — `create_map_object`** (단계별 1회씩, 공통 파라미터)
```
공통: width=32, height=32, view="low top-down", outline="selective outline", shading="basic shading"
seed   : "tiny planted seed mound, muted dark soil with faint blue spirit glow" (detail="low detail")
sprout : "small glowing blue spirit grass sprout, two thin luminous blades, soft blue glow"
mature : "harvest-ready glowing blue spirit grass, full luminous tuft clearly bigger than a sprout,
          bright blue spirit glow"
```

**지형 타일 — `create_topdown_tileset`** (Wang, lineless)
```
공통: view="low top-down", outline="lineless", shading="basic shading", tile_size=16x16
풀↔흙길 : lower_description="dirt path", upper_description="grass"
흙길↔밭흙: lower_description="dirt path", upper_description="tilled farm soil with furrows",
           lower_base_tile_id=<풀↔흙길의 dirt base id>   # ★체이닝으로 이음매 일치
```
→ 타일셋은 16타일 시트(64×64)로 나온다. **순수 타일**(corners 4개가 전부 같은 terrain)을 metadata의
   `bounding_box`로 16×16 크롭해 `tile_ground/path/soil.png` 추출(글루, ADR-0001 허용).

### 4.3 비용·시간 (실측)

- **총 13 generations / 핵심 7에셋** (트라이얼 무료 40 gen 중). 내역:
  - `create_character`(standard 4-dir) = **1 gen**
  - `animate_character`(walk template) = **방향당 1 gen → 4방향 4 gen**
  - `create_map_object` = **에셋당 1 gen** (혼령초 3 = 3 gen) ← 작물·소품의 정답 경로
  - `create_topdown_tileset` = **셋당 2~3 gen** (타일 2종 산출 → 타일당 ~1.25 gen)
- wall-clock: 캐릭터 ~3분, walk 4방향(병렬) ~4분, map_object/타일셋 ~90~100초. 폴링 대기 포함 세션 ~15분.
- **외삽 경고:** 트라이얼 40 gen은 스파이크용. P2.1~P2.3 벌크(NPC 4종 × 다방향/다애니 + 작물 다수 + 타일 다수)는 40 gen을 초과 → **유료 구독/크레딧 전환 필요**. gen당 단가는 위 내역으로 산정.

### 4.4 발견사항 (하네스·파이프라인에 반영 완료)

- **캐릭터 규격:** `size=32` 요청 → PixelLab이 캔버스를 ~1.4배 키워 **48×48 프레임** 출력. walk 템플릿은 **방향당 6프레임**. → `asset_preview.gd`의 `CHAR_FRAME=48×48`, `CHAR_FRAMES_PER_DIR=6`로 잠금. (16×32 타겟과 다름 — 벌크에서 Aseprite 다운스케일 또는 규격 재협의 필요.)
- **출력 포맷:** 캐릭터는 시트가 아니라 **방향별 개별 PNG**(south/north/east/west × 프레임). → 4행(down/up/right/left)×N열 시트로 합성하는 글루 필수.
- **작물 규격:** `create_map_object` 최소 캔버스 32px → 16 타겟과 불일치. 현재 32px 원본 사용(하네스는 바닥정렬로 수용). 벌크는 트림+다운스케일 단계 필요.
- **금지:** `create_1_direction_object`는 20~40 gen(트라이얼 몰살) → 절대 사용 금지. 작물·소품은 `create_map_object`.
- **타일셋 체이닝:** 인접 지형은 이전 타일셋의 base_tile_id를 `lower_base_tile_id`로 넘겨 이음매 일치.

### (참고) FAIL이었다면
손절 사다리(ROADMAP P2.0) — ① 스타일 완화 재스파이크 → ② 범위 축소 → ③ 폴백(외주/유료팩/제미나이 img2pixel는 PixelLab 호스트 변환이라 ADR 무방).

## 5. P2.1 캐스트 벌크 — 추가 레시피·발견 (2026-06-21)

플레이어 + 미호·옥자·바나·멜 5종을 `game/assets/characters/`에 도색. 미리보기는 `game/cast_preview.tscn`.

### 5.1 캐스트 비주얼 소스
- **원작 IP 일러스트가 진실의 원천.** 옥자·바나·멜·미호는 원작 컨셉카페 캐릭터 그림(제미나이 픽셀 일러스트)을 받아 그 외형을 영어 description으로 옮겨 `create_character`에 박았다(이미지 직접 입력 아님 — 텍스트 변환).
  - 미호: 검은 장발·흰 여우귀·**흰 저고리+노란 치마**·흰 여우꼬리·파란 여우불 (P2.0의 "노란 한복 전체"는 추측이라 폐기).
  - 옥자: 검은 마녀모자(빨간 깃털)·와인 곱슬 장발·안경·버건디 빅토리안 드레스.
  - 바나: 붉은 눈·송곳니·금발+검정 투톤·보라 고딕로리타.
  - 멜: 청록 만주 치파오·파란 술 만주모자·검은 단발.
- **플레이어 = 의도적 무채.** 원작 비주얼 없음 → 죄목("조용한 죄=빈칸") 기반 중립 디자인(남, 다크브라운 단발, 저채도 평상복). 후속 팔레트 스왑 베이스.

### 5.2 ★ hole-fill 후처리 (필수 — 안 하면 "속 비고 타일 비침")
PixelLab 캐릭터/워크 프레임은 **외곽선에 갇힌 투명 구멍**을 자주 남긴다(특히 정면 워크의 흰 꼬리 속, 미호 정면 9프레임에서 275px). 단순 alpha(투명/불투명) 검사로는 안 잡히고 **flood-fill로 "셀 경계와 연결된 외부 투명" vs "갇힌 내부 hole"을 구분**해야 보인다.
- 파이프라인: 시트 48px 셀마다 ① 경계에서 flood-fill로 외부 투명 마킹 → ② 남은 투명 = hole → ③ hole을 인접 불투명색으로 dilation 채움(흰 꼬리는 흰색으로 메워짐).
- 전 캐릭터 일괄 적용(미호 275·플레이어 3·옥자 2). **검증 = hole 0**.

### 5.3 워크 부위 일관성
- 가는 부위(꼬리·귀)는 **template walk가 프레임마다 크게 펄럭/출렁**여 깜빡인다 → **`mode="v3"` + `action_description`에 "single tail steady, no sway, ears fixed"**가 더 안정. v3는 방향당 9프레임(1 ref+8).
- **꼬리는 1개로 디자인**(description에 "exactly ONE single fox tail")하면 좌우 펄럭 원인 자체가 줄어든다(아홉 꼬리는 48px에서 무리).
- 상주 NPC(옥자·바나·멜)=rotation 4방향 1프레임 **idle**(애니 없음) / 움직이는 플레이어·미호=**v3 walk**.

### 5.4 Godot import 갱신 함정
PNG를 **같은 이름으로 교체**하면 Godot이 옛 import 캐시를 써 로드 실패/구버전 표시. → `assets/.../X.png.import` 삭제 + `.godot/imported/X.png-*` 삭제 후 `godot --headless --import` 재실행.

### 5.5 실행 글루
긴 절대경로를 셸에 복붙하면 줄바꿈으로 깨진다 → `game/run_cast.sh`(내부에서 `--path $PWD --resolution`)로 감싸 `bash <절대경로>` 한 토큰으로 실행. 캐릭터 시트 규약은 P2.0과 동일(48×48, 행=down/up/right/left), 단 워크 열 수는 시트 너비÷48로 자동 인식.

## 6. P2.2 작물 벌크 — 추가 레시피·발견 (2026-06-21)

피안화·영혼 호박 각 3단계 = 6 gen을 `create_map_object`로 도색, `game/assets/crops/`에 저장. 미리보기는 `game/crop_preview.tscn`(`bash game/run_crops.sh`). 혼령초(P2.0)까지 3작물 × 3단계가 한 화면에서 비교된다.

### 6.1 통한 프롬프트 (실제 호출값, §4.1 공통 파라미터 그대로 복제)
공통: `width=32, height=32, view="low top-down", outline="selective outline", shading="basic shading"`. **씨앗=low / 새싹=medium / 수확=high detail.** 작물 정체성(CONTEXT '저승 작물')을 description에 박고, **영혼빛 액센트 색을 작물마다 다르게** 줘 단계 구분 + 작물 구분을 동시에 만든다(혼령초=파랑·피안화=빨강·호박=주황).

```
피안화(彼岸花, 저승 상징 붉은 꽃):
 seed  : "tiny planted seed mound in dark furrowed soil, faint crimson red spirit glow seeping from the seed"
 sprout: "small young sprout with a thin dark-green stalk and a closed crimson flower bud beginning to form, faint red spirit glow"
 mature: "harvest-ready blooming red spider lily higanbana flower, distinctive long curled radial spidery petals
          and upward arching stamens, clearly a full vivid red flower much bigger than a sprout, crimson red spirit glow"
영혼 호박(Soul Pumpkin, 얼굴이 살짝 비치는 호박):
 seed  : "tiny planted seed mound in dark furrowed soil, faint warm orange spirit glow seeping from the seed"
 sprout: "small pumpkin vine sprout with a few broad green leaves and a tiny round green gourd starting to swell, faint orange spirit glow"
 mature: "harvest-ready large round orange pumpkin sitting on the ground, a faint ghostly spectral face softly glowing
          through the rind like a jack-o-lantern, clearly much bigger than a sprout, soft blue-white soul glow around the face"
```
공통 꼬리: `muted underworld palette, selective dark tonal outline not pure black, 2-3 tone shading, top-down`. **"clearly much bigger than a sprout"가 수확 단계 크기 점프를 만든다**(§4.1 혼령초 mature와 동일 트릭).

### 6.2 ★ 결과 PNG 다운로드 글루 (P2.0엔 미기록 — 이번에 확정)
`create_map_object`는 ID만 즉시 반환 → `get_map_object(id)`가 `status: completed`일 때 **`download:` URL**(`https://api.pixellab.ai/mcp/map-objects/<id>/download`)을 준다. MCP 이미지 블록은 디스크 저장 불가이므로 **그 URL을 curl로 받는다**(ADR-0001 허용 글루 = 받기·임포트):
```
curl -s -o assets/crops/<name>.png -H "Authorization: Bearer $PIXELLAB_API_KEY" \
  "https://api.pixellab.ai/mcp/map-objects/<id>/download"
```
키는 `.claude/settings.local.json`의 `env.PIXELLAB_API_KEY`에서 셸 변수로만 읽고 **절대 출력/커밋하지 않는다**. 결과는 32×32 RGBA(요청 그대로), 하네스가 바닥정렬로 수용 → 다운스케일 불요. **map object엔 §5.2 hole 없음**(flood-fill 검사 0 — 외곽선에 갇힌 투명 구멍은 캐릭터 워크 프레임 고유 문제).

### 6.3 파일명 = CropCatalog 영문 id 정렬
미래 T2.3 성장 로직이 `"res://assets/crops/%s_%s.png" % [crop_id, stage]`로 로드하도록 **파일 stem = `crops.gd`의 id**(`honryeongcho`·`pianhwa`·`yeonghon_hobak`) + `_seed/_sprout/_mature`. 혼령초 P2.0 산출(`honryeong_*`, 스파이크 폴더)도 `assets/crops/`로 옮기며 `honryeongcho_*`로 정규화(스파이크 `asset_preview.gd`가 참조하는 원본은 그대로 둠).
- **⚠️ seam:** 아트는 작물당 3프레임인데 `crops.gd` `stages`는 혼2/피3/호4. T2.3이 N 논리단계를 3 아트프레임에 매핑해야 한다(예: 마지막 프레임=수확가능 고정, 중간은 보간/반복). P2.2 범위 아님 — 기록만.

### 6.4 운영 발견
- **rate limit:** 6건을 한 번에 던지면 4건만 받고 2건 `rate limit exceeded`(~15–30s 후 재시도하면 통과). 벌크는 4건씩 끊어 던지는 게 안전.
- **서버측 실패:** 1건이 `[500] Out of CUDA memory`로 실패(내 파라미터 무관) → 동일 프롬프트 재생성으로 통과. 폴링 시 `failed` 상태를 반드시 확인하고 재던질 것.
- **비용:** 6 gen 사용(재시도 1 포함 실측 7 gen). 구독 Tier 1(2000 gen/주기)로 P2.0 트라이얼 경고(§4.3) 해소 — 잔량 충분.

## 7. P2.3 단계① 지형 오토타일 — 레시피·★Godot terrain 함정 (2026-06-21)

풀↔길↔밭 3지형을 **Godot terrain 전환**(corner 오토타일)으로. PixelLab Wang 타일셋 3세트를 공식 컨버터로 합쳐 `game/assets/tiles/combined_terrain.tres`(3 terrain) 생성, `main.gd`의 `_build_tileset`/`_paint_grid`가 이를 로드해 맵을 칠한다. 미리보기는 `tools/map_dump.gd`(렌더 없이 타일 배치를 PNG로 합성 — 헤드리스 GPU 불요).

### 7.1 통한 레시피 (실제 호출값)
- **3 전환 세트 = `create_topdown_tileset`**(`outline="lineless"`, `shading="basic shading"`, 16×16, `view="low top-down"`). base 체이닝으로 이음매 일치:
  - 풀↔흙길: `lower="dirt path"`, `upper="muted desaturated mossy grass, low saturation somber underworld graveyard green, ..."`, `lower_base_tile_id=<기존 dirt path base>`
  - 밭흙↔풀: `lower="tilled farm soil with furrows"`, `upper=<같은 풀 문구>`, `lower_base=<밭흙 base>`, `upper_base=<풀 base>` ★풀↔밭 직접 경계용(밭이 풀 한가운데 박혀 직접 닿음)
  - 흙길↔밭흙: P2.0 산출 재사용(`path` base 공유로 이음매 일치)
- **다운로드:** `get_topdown_tileset`이 준 `download_png`/`download_metadata`는 **302 redirect(backblaze)** → `curl -L` 필수(없으면 0바이트). 파일명은 컨버터 규칙 `*_metadata.json`/`*_image.png`로.
- **컨버터:** PixelLab 공식 `tools/pixellab_tileset_converter.gd`(MCP `pixellab://docs/godot/wang-tilesets`를 벤더링, output만 `res://`로). 인자 순서가 terrain id를 고정(0=길·1=풀·2=밭). 한 번에 3쌍 → corner peering bit 자동 설정된 단일 `.tres`.

### 7.2 ★ Godot terrain 함정 (이게 막혔던 핵심 — 다음 지형 작업에 그대로 적용)
1. **`set_cells_terrain_connect(cells, set, terrain, ignore_empty_terrains=false)`** — 4번째 인자 **반드시 false**. 기본 true면 빈 캔버스 첫 칠에서 모든 이웃이 empty라 제약이 사라져 **한 칸도 안 칠해진다**(by_source에 terrain 0칸). false라야 base가 매칭됨.
2. **모드 = `TERRAIN_MODE_MATCH_CORNERS`(1).** 컨버터 `.tres`는 `mode=0`(CORNERS_AND_SIDES)으로 나오지만 Wang은 코너만 설정 → side 미설정으로 매칭 깨짐. 로드 후 `set_terrain_set_mode(0, MATCH_CORNERS)` 강제.
3. **칠 순서:** 모든 지형 칸을 *풀 하나로 먼저* 단일 칠(연속 영역이라 검은 구멍 0) → 밭을 얹어 전환 → **단색(HOUSE/CAFE/WALL)은 맨 나중에 set_cell**. 단색을 먼저 깔면 그 terrain 없는 타일이 인접 풀의 corner 매칭을 깨 빈칸(검은 구멍)을 남긴다.
4. **1칸 폭 동선(길)은 base 직접 set_cell.** corner 전환은 양옆이 풀이라 길이 풀로 묻힌다 → `_terrain_base_atlas(TR_PATH)`로 또렷한 흙길 base를 직접 깐다(전환 없이). 넓은 밭은 terrain 전환 유지(경계 부드러움).

### 7.3 풀 채도 후처리 (P2.0 '하향 1순위' 이행)
`create_topdown_tileset` description만으론 풀이 여전히 순녹(#048e0c). `tools/desaturate_grass.py`(PIL, ADR-0001 허용 색보정 = Aseprite 보정)로 **녹색 hue(80–175°) 픽셀만** 채도×0.55·명도×0.82 → `#306033`(어두운 이끼). 흙(갈색)은 hue로 보존. 원본은 `*_raw.png`로 1회 백업(idempotent). 보정 후 컨버터 재실행 → import.

### 7.4 흙 상태(DRY/WET) 도트
`_build_field_tileset`이 단색 고랑 대신 **밭흙 terrain base를 추출**(`_extract_soil_base`)해 DRY=어둡게(×0.80)·WET=짙고 푸르게(×0.55 +blue)로 틴트. 무외곽선. 가운데 새싹 점은 성장 표시 그레이박스 유지(P2.2 작물 스프라이트 연결은 T2.3 seam).

### 7.5 비용·범위
- gen: 신규 2세트(풀↔길·밭↔풀) ≈ 5 gen. 흙 상태·풀 채도는 gen 0(파생·후처리).
- 단계 ① 범위 = 지형·흙만. **실내 바닥·벽·가구·장식(HOUSE/CAFE/WALL 단색 유지)은 단계 ②, 밤 라이팅은 단계 ③**(단계별 PR).

## 8. P2.3 단계② 실내 바닥·벽·가구·장식 — 레시피·★비용 함정 (2026-06-21)

실내 단색 3종(HOUSE/CAFE/WALL)을 도트 타일로, 가구·장식 6종을 `create_map_object`로. `main.gd`의 `_build_tileset`이 단색 fill 대신 16×16 PNG를 SOLID source(id 1)에 blit하고, 가구는 손님·잡귀처럼 `_draw_props()`가 캐릭터 *아래*에 바닥정렬로 그린다(`PROP_LAYOUT` 데이터). 충돌은 WALL 타일만 — 가구는 순수 장식(art 패스라 새 시스템·이동 변화 금지).

### 8.1 ★ 비용 함정 — `create_tiles_pro`는 ~20–40 gen (피하라)
실내 바닥/벽 같은 **단일 면 타일**을 만들 때 `create_tiles_pro`를 먼저 시도했으나 호출 직후 `cost: ~20-40 generations`로 떴다 — §4.4가 금지한 `create_1_direction_object`(트라이얼 몰살)와 같은 등급. **생성 중 삭제는 막혀(이미 과금) 환불 불가.** 16타일(variations 16)을 한 번에 채색하느라 비싸다.
- 함정: cost는 호출 *결과*에만 떠 사전 회피 불가 → **`create_tiles_pro`는 비용을 알고 쓸 때만**(작은 변형 다발이 정말 필요할 때). 단일/소수 타일엔 부적합.
- 부산물: 이미 과금됐으니 그 16타일에서 쓸 것만 골라 건졌다 — `tile_2`=청회 판석(집)·`tile_4`=앰버 판자(카페)·`tile_13`=어두운 벽돌(벽). 16×16 flat(`square_topdown`+`tile_view=top-down`+`depth_ratio=0` → 깊이 없는 정면 타일). 4×4 반복 이음매 점검 후 `assets/tiles/{house_floor,cafe_floor,wall}.png`로 저장.
- 폴백(저비용)으로 같이 띄운 `create_topdown_tileset`(transition_size=0, lineless, 16px) ≈ 2.5 gen는 집·카페 바닥 base 2장을 정상 산출 — **다음에 단일 면 타일이 필요하면 이 경로**(base_tile_id로 추출). 이번엔 tiles_pro가 벽까지 줘서 그걸 썼다.

### 8.2 가구·장식 = `create_map_object` (§6.1 작물과 같은 경로)
- 공통: `view="high top-down"`, `outline="selective outline"`(전경 컨벤션), `shading="basic shading"`, 꼬리 `muted underworld palette, selective dark tonal outline not pure black, 2-3 tone shading`.
- 6종: 카페 카운터·스툴·선반 / 집 침대(32×64) / 소울 등불(영혼불 블루) / 혼령초 화분. 침대만 height 64로 1×2칸, 나머지 32×32.
- 다운로드: §6.2 글루 그대로(`/mcp/map-objects/<id>/download` curl). **32px 원본 → `tools/downscale_prop.py`(PIL NEAREST 2:1)로 16×16(침대 16×32)** 다운스케일 = ADR-0003 16px 격자·픽셀밀도 일관(원본 32px는 격자보다 2배 고밀이라 그대로 그리면 안 됨). 원본은 `*_raw.png`로 백업 커밋(①의 terrain _raw와 같은 결).
- **rate limit:** §6.4대로 4건씩 끊어 던졌다(4 + 2).

### 8.3 통합·검증
- `_build_tileset`: 단색 fill 루프 → `load(SOLID_TEX[id]).get_image()` blit(아틀라스 결·WALL 충돌·칠 순서 불변 → `_paint_grid`·`_set_tile`·세이브 손 안 댐).
- `_draw_props()`: `_draw` 맨 앞에서 호출(자식 노드=캐릭터·손님이 위에 올라옴). 좌석 스툴(y7) 위에 손님 박스가 자연히 덮인다. 배치(직원 y5/카운터 y6/스툴 y7/통로 y8)는 바 레이아웃으로 읽힌다.
- `tools/map_dump.gd`에 `PROP_LAYOUT` 합성 추가(GPU 없이 가구까지 한 장 확인).
- 검증: 헤드리스 import·부팅(120f) 클린, 회귀 7종(cafe·night·bana·weave·milestone·npc·cafe_margin) PASS, 맵 덤프 배치 육안 확인.

### 8.4 비용·범위
- gen: 가구·장식 6 + 바닥/벽(tiles_pro ~20–40 함정 1회 + 폴백 topdown ~2.5). **교훈: 단일 면 타일은 topdown_tileset base, tiles_pro 금지.**
- 단계 ② 범위 = 실내 표면·가구·장식만. **밤 라이팅(CanvasModulate/Light2D)은 단계 ③.** 카페 floor 앰버는 다소 밝음(아늑한 의도) → 거슬리면 P2.7 톤 패스에서 하향.
