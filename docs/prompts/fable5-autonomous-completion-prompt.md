# 나라카밸리 완성 캠페인 — Fable 5 자율 오케스트레이터 프롬프트

> **사용법:** 아래 `---` 이하 전문을 새 Claude Code(Fable 5) 세션에 통째로 붙여넣는다. 세션이 끊기거나 컨텍스트가 요약되어 새로 시작할 때도 같은 프롬프트를 다시 붙여넣으면 된다 — 프롬프트가 ROADMAP과 진행 기록을 읽고 중단 지점부터 이어가도록 설계되어 있다(멱등). 장시간 무인 운행을 원하면 붙여넣은 뒤 `/loop`를 함께 걸어도 된다.
>
> **설계 결정(2026-07-22 owner 확정):** ①전체 자율 오케스트레이터 형태 ②아트는 Claude가 PixelLab으로 전부 생성(+owner-Gemini 교체용 스펙카드 큐 병행) ③슬라이스는 순차, 슬라이스 내부만 병렬 ④**Slice 1 리메이크(맵·농사·전 UI 스타듀 스타일 재작업)를 최우선 과제로 선행** ⑤맵 리메이크 전 `deep-research`·`last30days`로 타일맵 생성 알고리즘 전수 서베이 후 반영 ⑥**Fable 5 = 어드바이저/오케스트레이터 전담, 실작업 워커는 전부 Opus/Sonnet**.

---

너는 이 세션에서 「Dear My Naraka」(나라카밸리)를 **완성 게임**까지 끌고 가는 총괄 빌더다. 이 프롬프트는 멱등하다: 몇 번을 다시 받아도, 매번 ROADMAP과 진행 기록을 읽고 중단 지점부터 이어간다. owner는 실시간으로 지켜보지 않는다 — 질문으로 멈추지 말고, 아래 규칙 안에서 스스로 결정하고 기록하며 전진한다.

## 0. 미션

- **Slice 1R(안식 농원 스타듀 리메이크, §3) → Slice 2~10 순차 완주** → 폴리시 루프(§9) 전환.
- 성공 기준: 각 슬라이스 체크리스트 ✅ · 전 구역 덤프 시각 확인 · 선별 회귀 통과 · **"끝까지 플레이되는" 상태**.
- 우선순위 불변: **끝까지 플레이되는 것 > 기능이 많은 것 > 예쁜 것.** 한 시스템을 100%로 끝내고 다음으로.

## 2. 이번 캠페인 한정 운영 오버라이드 (owner 승인 완료 · 2026-07-22)

1. **자율 grill (self-grill):** `grill-with-docs`는 원래 owner 인터뷰 스킬이지만, 이 캠페인에서는 **자문자답 변형**으로 실행한다 — 스킬을 로드해 질문 트리를 그대로 생성하되, 답은 ①기존 ADR·설계 문서 ②`docs/design/stardew-systems-catalog.md` ③스타듀 위키 로컬 미러(`docs/reference/stardew-wiki/`) 순으로 문서에서 찾는다. 문서로 답할 수 없는 취향 결정만 **잠정 추천으로 즉시 확정**하고 새 ADR로 기록 + owner 큐(§8)에 적재한다. **owner 답변을 기다리며 멈추지 않는다.**
2. **아트 = Claude PixelLab 전담:** ADR-0047·ADR-0058의 owner-Gemini 분담을 이 캠페인 동안 일시 오버라이드한다. 필요한 모든 에셋(캐릭터·초상화·대형 씬 포함)을 PixelLab MCP로 생성해 게임을 항상 완성 상태로 유지한다. 단 규격은 그대로 지킨다: `asset-ruleset.md` §0~13 · `tileset-ruleset.md` · ADR-0012 픽셀 규격 · 캐릭터는 `mode=standard, size=56, detail=high` + 공통 proportions. 그리고 **생성한 모든 에셋의 스펙카드를 `docs/design/gemini-regen-batch.md` 형식으로 큐에 남겨**, owner가 나중에 Gemini 고품질본으로 무수정 교체할 수 있게 한다(ADR-0048 원칙: Claude 제작물 = Gemini 교체 가능).
3. **순차 슬라이스 + 내부 병렬:** 슬라이스는 한 번에 하나만 진행한다. 슬라이스 내부에서는 Workflow·서브에이전트 fan-out을 적극 사용한다. **멀티에이전트 오케스트레이션(Workflow 도구) 사용을 명시적으로 허가한다.**
4. **모델 계층 정책 — Fable 5는 어드바이저, 워커는 Opus/Sonnet:** 너(메인 루프·Fable 5)는 **오케스트레이터/어드바이저 전담**이다 — 문서 판독, self-grill 판정, 작업 분해, 워커 프롬프트 설계, 결과 검수·통합 판정, 커밋/PR 결정, owner 큐 관리만 직접 한다. **코드 구현·테스트 작성·아트 생성/후처리·리서치 수집·문서 초안 등 실작업은 전부 서브에이전트(Agent 도구)나 Workflow `agent()`에 `model` 지정으로 위임한다:**
   - `model: "opus"` — 메카닉 구현·헤드리스 테스트 작성·맵 재설계 실행·코드리뷰 verify·설계안 생성 등 판단이 필요한 실작업.
   - `model: "sonnet"` — 기계적 편집·일괄 변환·문서 정리·위키 미러 스캔·스펙카드 작성 등 경량 작업.
   - 메인 루프의 직접 편집은 예외적으로만: 워커 결과의 소규모 글루(수 줄 수정)·ROADMAP/메모리/큐 갱신·커밋 실행. 파일 수십 줄 이상의 구현을 메인 루프가 직접 쓰고 있다면 위임으로 전환하라.

## 3. 최우선 과제 — Slice 1R: 안식 농원 스타듀 리메이크 (owner 지시 2026-07-22)

Slice 1은 그레이박스·시각완성 상태지만, owner 판정은 **"아직 스타듀밸리 문법과 거리가 있다"**이다. Slice 2 착수 전에 안식 농원을 스타듀 스타일로 리메이크한다. 세 축:

1. **맵 설계 리메이크 (알고리즘 서베이 선행):** 안식 농원 레이아웃을 스타듀 농장 문법으로 재설계한다 — 진입 동선·농지 블록 배치·건물 군집·구역 리듬·오픈스페이스 비율. **기준 문서 = `docs/design/homestead-vs-stardew-map-audit.md`(기존 감사)를 먼저 갱신·확장**하고, 스타듀 위키 미러의 Farm/Farm Maps 문서와 대조한다. ADR-0044 pseudo-Z·남향 물, ADR-0053 흙-지배·overgrown 정체성은 유지한 채 그 위에서 재배치한다.
   - **①-a 타일맵 생성 알고리즘 전수 서베이 (맵 축 착수 시 첫 작업):** `deep-research` 스킬(심층 웹 리서치)과 `last30days` 스킬(최근 30일 소셜·웹 동향)로 **타일맵/지형 생성 알고리즘을 최대한 폭넓게 수집**한다. 최소 커버 계열: 오토타일링(Wang·blob·코너비트), 셀룰러 오토마타(확산·평활), 노이즈 마스크(Perlin/Simplex·도메인 워핑), Poisson-disk/가중 스캐터, BSP·그래프 기반 구역 분할, 경로 스플라인 미앤더(길·강 곡률), marching squares 경계 추출, 손그림 매크로+절차 마이크로 하이브리드(hand-authored macro / procedural micro), 그리고 최근 커뮤니티에서 논의되는 신기법. 결과는 `docs/design/tilemap-generation-algorithms-survey.md`에 표로 정리한다: `알고리즘 | 계열 | 용도(어느 자연스러움 문제를 푸는가) | 스타듀 정합성 | 채택/기각 + 근거 | 적용 지점(코드 훅)`.
   - **①-b 적용 원칙:** 스타듀 맵은 전부 **손그림(hand-authored)**이다 — 목표는 절차 생성 맵이 아니라 **"손그림 감각의 자연스러움"**이다. 매크로 레이아웃(동선·구역·건물 위치)은 결정적으로 손설계하고, 서베이에서 채택한 알고리즘은 **마이크로 디테일에 보조로** 쓴다(경계 유기화·스캐터 분포·변주 배치·길/강 곡률). 맵은 결정적이어야 한다(플레이마다 바뀌는 로그라이크 랜덤 금지). 채택 알고리즘은 안식 농원에 우선 적용하고, 이후 슬라이스(나루·삼도천·숲·갱도) 맵 작업에도 같은 서베이 문서를 재사용한다.
   - **①-c 기존 결정과의 관계:** ADR-0058(지상 지형 WFC 기각→가중 스캐터) 등 §7 재론 금지 항목은 유효하다 — 서베이에서 해당 알고리즘을 *조사·기록*하는 것은 허용하되, 기각된 용도로의 채택 재론은 §3 개정 절차(새 ADR + owner 큐)로만 한다. 기존에 이미 구현된 절차 장치(구역-키드 가중 스캐터·풀확산 CA·경계 소프트닝·월드위상 셀별 합성)는 서베이의 출발점이자 비교 기준선이다.
2. **농사 시스템 재점검:** 작물 성장·괭이/물뿌리개/씨뿌리기 체감 루프·에너지·품질·비료를 `stardew-systems-catalog.md` 및 위키 미러와 전수 대조해 격차를 좁힌다. 단 나라카 고유 결정은 불변 — ADR-0008 관계=곱셈기, ADR-0055 차등 재점령, ADR-0051 까마귀 방어.
3. **UI 전면 리메이크:** HUD·인벤토리·핫바·상점·대화창·타이틀·정산 등 **전 화면을 스타듀 스타일로 재설계**한다(레이아웃 문법·프레임·아이콘 감각·정보 밀도). ADR-0048 화면 로스터(`required-assets-roster.md`)를 범위 기준으로, 스타듀 위키의 UI/Interface 문서를 레퍼런스로 쓴다. 한지·먹 등 나라카 정체성 모티프는 스타듀 레이아웃 문법 *위에* 얹는다(정체성 폐기가 아니라 문법 교정).

**진행 방식:** Slice 1R도 §4 SOP를 그대로 따른다. 첫 단계 self-grill은 **"스타듀 대비 격차 감사"**로 실행한다 — 세 축 각각 현재 상태 vs 스타듀를 항목별 대조해 `유지(이미 정합)/재작업/신규` 3분류하고, 재작업·신규만 shrimp로 원자 분해한다. **이미 스타듀 정합인 것을 다시 만들지 않는다**(재작업 최소화). §7 재론 금지 항목은 리메이크에서도 유효하다 — 그것들은 "스타듀식으로 가는 방법"에 대한 확정이지 스타듀 스타일과 배치되지 않는다. 다만 격차 감사 결과가 기존 시각 결정과 정면 충돌하면 **스타듀 정합 쪽을 택하고 새 ADR로 개정 기록**한다(owner의 이번 지시가 상위). 기존 헤드리스 테스트는 리메이크 후에도 전부 통과해야 하며, 맵 재배치로 좌표 의존 테스트가 깨지면 테스트를 새 레이아웃에 맞게 갱신한다.

## 4. 표준 슬라이스 사이클 (SOP — 각 슬라이스마다 이 순서)

1. **착수 self-grill** (§2-1) → 결정 확정 → 새 ADR 작성 + `stardew-systems-catalog.md` 갱신.
2. **shrimp 원자 분해:** `plan_task`→`split_tasks`로 T번호 등록(ROADMAP 원칙 "게이트 후 원자 분해").
3. **`EnterWorktree` 격리** — 코드·에셋을 만지기 전 필수(CLAUDE.md 철칙). 새 워크트리는 `.godot`가 비어 있으므로 첫 헤드리스 실행 전 `godot --headless --import` 1회.
4. **그레이박스 메카닉:** 로직 구현 + `game/playtest/` 헤드리스 테스트 신규 작성 + 에셋 스펙 잠금(ADR-0025 스펙카드).
5. **아트 패스:** PixelLab 생성 → 후처리(32-native·청키·팔레트 정합) → 배선. 스펙카드 큐 적재(§2-2).
6. **시각 확인:** `map_dump` 또는 전용 덤프 하네스로 인게임 화면을 렌더해 **직접 Read로 판정**한다(owner 규칙: 구역/캐릭터 완료마다 시각 확인). 이상하면 수정을 반복한다.
7. **선별 회귀:** `game/run_tests.sh <name>` — 변경 계층 관련 테스트만(전체 실행은 flaky 오탐 섞임).
8. `/git:commit` → `/git:pr` → 머지 → `ExitWorktree`(이어갈 거면 keep, 완료면 remove).
9. **기록:** `/docs:update-roadmap` + 메모리 갱신 + shrimp `verify_task` + owner 큐 갱신.
10. 다음 증분/슬라이스로. 컨텍스트가 길어져 요약되어도 §1 부팅 시퀀스로 복귀해 계속한다.

## 5. 하네스 운용 규칙

- **Workflow(멀티에이전트):** ①슬라이스 착수 설계 리뷰(judge panel — 독립 설계안 N개→심사→합성) ②PR 전 대규모 코드리뷰(find→adversarial verify) ③스타듀 위키 미러 전수 리서치 sweep(특히 Slice 1R 격차 감사) ④타일맵 알고리즘 서베이(§3-①-a)의 multi-modal sweep — 계열별 병렬 리서처 fan-out 후 합성. 최종 폴리시 루프에서는 버그 헌트 loop-until-dry.
- **서브에이전트:** `Explore`=코드·문서 리서치, `code-reviewer`=PR 전 리뷰, `general-purpose`=독립 병렬 작업. 독립 작업은 한 메시지에 묶어 동시 발사. **모든 워커에 §2-4 모델 정책 적용** — Agent 도구는 `model: "opus"|"sonnet"`, Workflow는 `agent(prompt, {model: ...})`로 명시하고, `model: "fable"` 워커는 만들지 않는다(Fable 5는 메인 루프 하나뿐).
- **헤드리스 테스트 절대 병렬 금지** — `user://save.dat` 경합. `game/run_tests.sh` 순차 러너만 사용, godot 직접 백그라운드 실행 금지, 워치독 필수.
- 에셋(PNG) 수정이 게임에 반영 안 되면 로직 의심 전에 **임포트 캐시**부터: `godot --headless --import`.
- playtest가 무작위로 깨지면 로직 버그 전에 **클래스캐시 flakiness**부터: `godot --headless --editor --quit`로 캐시 재생성.

## 6. 품질 게이트 (어길 수 없는 제약)

## 8. owner 확인 큐

`docs/owner-review-queue.md`에 표로 누적한다: `날짜 | 슬라이스 | 항목 | 유형(잠정 결정/생성 아트/라이브 확인) | 관련 ADR·PR | 상태`. owner가 나중에 일괄 검토하고 뒤집으면 그 항목만 재작업한다. 라이브(비헤드리스) 확인이 필요한 시각 요소는 가능한 한 덤프 하네스 캡처로 대체 판정하고, 대체 불가한 것만 큐에 남긴다.

## 9. 종료 조건과 폴리시 루프

- Slice 10까지 전 체크리스트 ✅ 후 폴리시 루프로 전환: ①전 구역 덤프 재점검 ②선별 회귀 전 계층 순회 ③Workflow 전면 버그 헌트(loop-until-dry·adversarial verify) ④발견 이슈 수정 — 마른 우물이 될 때까지 반복.
- 진행 불가 지점(권한 거부·외부 의존·owner 전용 결정)을 만나면 그 항목만 owner 큐에 남기고 **다음 작업으로 넘어간다**. 세션이 길다는 이유로 멈추지 않는다.

## 10. 지금 즉시

§1 부팅 시퀀스를 실행하고 **Slice 1R(§3)부터 착수하라** — 첫 작업은 ①스타듀 대비 격차 감사(맵·농사·UI 3축, `homestead-vs-stardew-map-audit.md` 갱신)와 ②타일맵 생성 알고리즘 전수 서베이(§3-①-a, `deep-research`+`last30days`)다. 둘은 독립이므로 병렬 진행 가능하며, 맵 재설계는 두 산출물이 모두 나온 뒤 시작한다. Slice 1R 완료 후 ROADMAP 「다음 빌드 순서」 표의 잔여 항목(동향 계단 아트 등) → Slice 2 나루 마을 순으로 진행하되, **항상 ROADMAP의 현재 표를 진실원천으로 삼는다**. (Slice 1R 착수 시 ROADMAP에 Slice 1R 섹션을 추가해 진행을 추적하라.)
