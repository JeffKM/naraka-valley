# ① 디자인 패턴 (서버 영역)

> **주의:** "손님 타입별 만족도 로직"은 게임플레이 = **클라(Godot) 영역**이라 서버 포폴 항목이 아니다.
> 서버에서 패턴이 진짜 필요한 곳으로 재배치한다.

선생님 평가의 핵심은 패턴 이름이 아니라 **"왜 썼나(트레이드오프)"**.
그리고 **남용 금지** — 진짜 로직이 갈라지는 케이스 3개+ 있을 때만 패턴을 쓴다.

> 🔁 옛 설계는 두 패턴을 웹샵/결제(상품 타입별 지급·주문 상태머신)에 묶었으나, 웹샵을 들어냈다(ADR-0003).
> 두 패턴은 *버리지 않고* 세이브 정산 파이프라인의 진짜 자리로 이사한다.

## Strategy — 진행 종류별 검증·반영 로직 (`DeltaApplier`)

세이브 동기화 시 클라가 올리는 진행분은 종류마다 검증 규칙·권위 반영 방식이 다름:

| 진행 종류 | 검증·반영 동작 |
|---|---|
| 재화 획득 | 획득 상한·소스 룰 검증 후 권위 재화 가산 |
| 작물 수확 | 경작 면적·시간 기반 상한 검증 후 인벤토리 반영 |
| 카페 성장도 | 활동 누적 룰 검증 후 성장 지표 반영 |
| 도감 수집 | 신규 종 유효성 검증 후 수집 집합 갱신(→ 수집률 랭킹) |

```java
public interface DeltaApplier {
    void apply(Account account, SyncDelta delta); // 검증 + 권위 반영
}
// GoldDeltaApplier, HarvestDeltaApplier, CafeGrowthApplier, CollectionApplier ...
// 새 콘텐츠 종류가 추가돼도 정산 코어를 안 건드리고 전략만 추가 (OCP)
```

→ 진행 종류가 4개+라 "**진짜 로직이 3개+ 갈릴 때만 패턴**" 원칙 충족. *재화 검증 ≠ 도감 검증*이라 분기가 진짜다.

## State — 세이브 정산 상태머신 (`SyncSession`)

```
READY ──검증──> VALIDATED ──반영(tx)──> APPLIED ──후속완료──> CONFIRMED
  │                 │                       │
  └──검증실패──> REJECTED          └──부분실패──> ROLLED_BACK(보상)
```

- 각 상태에서 허용된 전이만 캡슐화 → 잘못된 전이(예: ROLLED_BACK→CONFIRMED) 차단
- 정산 도메인의 무결성을 코드로 강제 (상세 `04-save-settlement-pipeline.md`)

## 그 외 (남용 않는 선에서)

- **랭킹 산정 정책** → Strategy (카페 성장도순 / 수집률순 / 주간)
- **정산 완료 후처리(랭킹 ZSET 갱신·공유카드 async·감사로그)** → Spring `ApplicationEvent`(Observer)로 디커플링 + outbox 보상(`04`)

## 포폴 문장화

- "8개 도메인 분기를 N개 패턴으로 해결하고 **선택 이유를 ADR로 기록**"
- 패턴마다 *대안과 트레이드오프*를 적어 "패턴을 위한 패턴" 역공을 차단
