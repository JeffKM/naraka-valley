# ③ I/O 최적화 — 세이브 동기화 + 읽기 캐싱

> **재확인:** 서버는 실시간 게임상태를 보유하지 않는다(클라 오프라인).
> 따라서 "농작물 매 초 write를 Redis로 막는다"는 **틀린 프레임**(성장은 클라 lazy 계산).
> 서버의 진짜 I/O는 ⓐ 세이브 동기화 시 대량 write, ⓑ 랭킹·상점 읽기, ⓒ 이미지 생성이다.

## ⓐ 세이브 동기화 — write 최적화

세이브/로그인 시 클라가 대량 데이터(인벤토리 100+슬롯, 재화, 카페 성장도)를 push.
이걸 **행마다 INSERT하면 부하 폭발** → 벌크 처리.

- **JDBC 배치 / `saveAll` + `hibernate.jdbc.batch_size`** — 슬롯 일괄 upsert
- **Spring Batch** — 주기적(예: 5분) 영속화 or 대량 동기화 잡
- **서버 검증 후 write** — 클라가 보낸 재화/아이템을 그대로 믿지 않고 규칙 검증(치트 방지)

### 미정 포크: 전체 스냅샷 vs 델타

| 방식 | 장점 | 단점 |
|---|---|---|
| **전체 스냅샷** (추천) | 단순·안전, 정합성 쉬움 | 매 저장 전체 직렬화 → write 큼 |
| 델타(변경분만) | write 효율 | 변경 추적·충돌·정합성 복잡 |

→ 추천: **전체 스냅샷 + 서버검증 + JDBC 벌크**로 시작, 델타는 측정 후 최적화.

## ⓑ 읽기 캐싱 — Redis

| 대상 | 방식 |
|---|---|
| **랭킹**(카페 성장도·수집률) | Redis **Sorted Set**(ZADD/ZRANGE) — 실시간 순위, DB 부담 0 |
| 도감/아이템 정의 등 정적 목록 | `@Cacheable` + Redis — 자주 안 바뀜 |

## ⓒ 비동기 / 기타

- **공유 카드 이미지 생성** — 무거운 작업은 `@Async`/큐로 분리 (pitch-ac OG 라우트와 같은 발상)
- **정산 후속 반영(랭킹 ZSET·공유카드)** — **outbox 패턴**으로 권위 반영과 한 커밋에 묶고 비동기 멱등 릴레이(`04`) — 실패 시 재시도/보상
- **커넥션 풀(HikariCP)** 튜닝 — 동기화 트래픽 몰릴 때 풀 고갈 방지

## 측정 (진부함 탈출)

- 벌크 적용 전/후 write 시간 비교표

## 면접 대비 — 유실 시나리오

> "Redis나 서버가 동기화 전에 죽으면 데이터 유실은요?"

- Redis는 **읽기 캐싱 용도**라 유실돼도 DB에서 재구성(원천 아님)
- write는 동기화 시점 트랜잭션으로 DB 직행 → 캐시 유실과 무관
- 세이브 자체 실패 대비: 클라 측 마지막 성공 세이브 보존 + 재시도
