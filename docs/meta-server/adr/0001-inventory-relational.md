# 인벤토리를 관계형 테이블로 모델링

유저 인벤토리(슬롯·아이템·커스텀 옵션)를 JSON 직렬화 컬럼이 아니라 관계형 테이블(user-inventory-slot-item-option)로 저장. **수집률/도감 랭킹의 크로스-유저 집계**·서버 검증·아이템별 쿼리를 가능하게 하기 위함.

> 🔁 **보정(ADR-0003):** 최초 근거는 "거래소"였으나, 거래소는 싱글플레이에 없는 가짜 기능이라 제거됐다.
> 관계형의 주력 근거는 **수집률/도감 랭킹의 크로스-유저 집계(GROUP BY)** 로 재정초한다 — 이건 JSON 블롭으론
> 전 유저 스캔이 필요해 관계형이 *필요*가 된다. N+1 트리거(세이브 동기화 로드)는 불변.

## Considered Options

- **JSON 한 컬럼에 직렬화** — 읽기/저장 단순, 게임 세이브엔 흔함. 단 아이템별 쿼리·**크로스-유저 집계(수집률 랭킹) 불가**, 서버가 개별 아이템을 검증/조작하기 어려움. N+1이 아예 없음(최적화 스토리도 없음)
- **관계형 테이블** (채택) — 수집률 랭킹 집계·서버검증·아이템별 쿼리 가능. 대신 조회 시 N+1이 필연 발생 → fetch join/batch로 최적화
- **하이브리드(핵심만 관계형 + 잡동사니 JSON)** — 현실적이나 두 모델 경계 설명이 복잡, 초기엔 과설계

## Consequences

- 수집률 랭킹 집계·아이템 단위 검증·통계가 자연스러워짐(치트 방지의 토대)
- 조회 시 N+1 발생 → `02-inventory-nplus1.md`의 fetch join/batch 최적화가 정당화됨(플래그십 스토리의 전제, ADR-0003)
- 세이브 동기화 시 다수 행 write → JDBC 벌크/Spring Batch 필요(`03-io-optimization.md`)
- JSON 대비 스키마 변경(새 아이템 속성) 비용↑ → ITEM_OPTION의 key-value로 일부 흡수
