# 나라카 메타 서버 — 아키텍처 & 개념 매핑

**새 Spring Boot 프로젝트.** 내가 Godot으로 만드는 게임 **naraka-valley**(저승 컨셉카페 세계관의
스타듀밸리형 타이쿤)의 **데이터 무결성·커머스·랭킹을 책임지는 메타 서버.**

> **한 줄 스토리:** "좋아하는 서브컬처 컨셉카페를 모티브로 타이쿤 게임을 Godot으로 직접 개발했고,
> 이 게임의 데이터 무결성과 트래픽을 감당하기 위해 Spring Boot로 메타 서버를 구축했다."
> → 기획력 · 실행력 · 기술 깊이 삼박자.

## 왜 싱글 게임에 서버가 필요한가 (치트 방지)

스타듀류는 본질이 오프라인 싱글이다. 모든 데이터(돈·인벤토리·카페 성장도)가 클라(Godot)에만
있으면 **메모리 조작으로 재화 치트·아이템 복사**가 가능하다.
→ **핵심 데이터·상점·결제는 서버가 검증/관리**한다. 이것이 메타 서버의 존재 이유.

## 서버 권위 범위 — "메타 계층만" (의식적 결정)

전부 서버 권위(온라인 강제)는 장르를 깬다(오프라인 플레이 불가). 그래서 권위를 **메타 계층에만** 둔다.

| 영역 | 주체 | 이유 |
|---|---|---|
| 농사·이동·손님·성장(lazy 계산) | **클라(오프라인)** | 스타듀 장르의 오프라인 경험 보존 |
| 클라우드 세이브·재화·인벤토리·랭킹 | **서버 권위** | 치트 방지 — 돈 되는 데이터 |
| 세이브 동기화 시 정산/지급 | **정산 파이프라인(서버)** | exactly-once 정합성 |

> 📌 이 범위 결정은 ADR-0002 참조. 커머스(거래소·결제 웹샵) 제거 및 정산 파이프라인 이전은 ADR-0003.
> 되돌리기 비싸고(클라/서버 책임 재분배), 트레이드오프가 분명함.

## 3블록 구조

```
[Godot 클라 (naraka-valley)]   오프라인 게임플레이: 농사·이동·손님·성장(lazy)
        │  로그인/세이브 시 동기화 (REST, 서버 검증)
        ▼
[나라카 메타 서버 (Spring Boot)]  권위 검증: 클라우드세이브·재화·인벤토리·랭킹
        │  세이브 동기화 시 정산(멱등·상태머신·outbox 보상)
        ▼
[DB(권위) + Redis(랭킹 ZSET) + async(공유카드)]  outbox로 트랜잭션 경계 넘는 분산 정합성
```

## ⚠️ 설계 주의 — "메타 계층만 권위"의 숨은 결과

서버는 **실시간 게임상태를 보유하지 않는다**(클라 오프라인). 이로 인한 함정 2가지:

1. **①③ 예시는 서버용으로 재배치 필수.** 원래 들었던 "손님 만족도 로직(①)", "농작물 매 초 성장(③)"은
   게임플레이 = **클라 영역**이다. 서버 포폴 항목으로 쓰면 "그건 클라에서 돌잖아요?" 역공.
   → ① = 정산/검증/랭킹 서버 로직으로, ③ = 세이브 동기화/캐싱으로 재배치.
2. **Redis는 "실시간 write-behind"가 아니라 "읽기 캐싱".** 서버가 실시간 데이터를 안 가지므로
   write 최적화는 *동기화 시점 벌크*, Redis는 *랭킹·상점 읽기 캐싱*에 쓴다. (`03` 참조)

| "싱글 게임인데 결제/거래소/선착순이 왜?" | 가짜 명분이라 제거(ADR-0003). 결제 엔지니어링은 세이브 정산으로 이전 |
| "외부 PG 없으면 보상 트랜잭션도 없잖아요?" | DB↔Redis↔async의 정합성 경계를 **outbox**로 보상 — 분산 정합성 무대 |
| "농작물은 timestamp로 계산하면 되잖아요?" | 맞음. 성장은 클라 lazy 계산. 서버 I/O는 세이브 동기화/랭킹 |
| "싱글 가방인데 JSON 말면 N+1 없는데?" | **수집률 랭킹의 크로스-유저 집계** 위해 관계형 채택(ADR-0001) |

| "유저 0명인데 대량 트래픽?" | k6/nGrinder **부하테스트로 X TPS·lost update 0건 검증**(측정값) |
