# 환경 아트 32px native 복귀 — AI 생성 소스는 16px 축소로 뭉개짐 (ADR-0049 supersede)

> **Status:** accepted (2026-07-04). **supersedes [ADR-0049](./0049-environment-16px-logical-stardew-grain-supersede-0013.md)** → 순효과 = [ADR-0013](./0013-environment-art-32px-native.md)의 **32px native 규격 복귀**(단 근거는 새로움).

## 맥락

[ADR-0049](./0049-environment-16px-logical-stardew-grain-supersede-0013.md)는 같은 날(2026-07-04) "스타듀 청키 그레인"을 노려 환경 아트 논리 해상도를 32→**16px**로 내렸다(정수 스케일·`TILE=32` 불변, 객체는 `chunkify_asset.py` ÷2→×2).

그 직후 **debris 3종(이승의 미련·업화석·석화 고목)의 1칸 아트를 생성**하며 owner가 실제 스타듀 스크린샷과 대조하니, **우리 16px 산출물이 스타듀보다 뭉개지고 덜 선명**했다. 원인을 파고든 결과:

- **스타듀 16px = 손그림 네이티브.** 작가가 16px 격자에 한 픽셀씩 의도적으로 배치하고 손으로 안티에일리어싱·셰이딩을 넣는다 → 같은 16px라도 깨끗하고 부드럽다.
- **우리 16px = AI 축소본.** PixelLab/Gemini가 64px로 생성한 그림을 16px로 다운스케일(BOX)+양자화하면 디테일이 뭉개진다([ADR-0047] Gemini·[ADR-0001] AI 생성 전제). **"손그림 네이티브 16px" ≠ "AI 축소 16px".**
- **비교 검증(`_RES_COMPARE`, 2026-07-04):** 같은 debris를 16/24/32 논리로 렌더 → **32가 스타듀 스크린샷에 최근접**(잎 결·바크·바위 면이 살아남). AI 소스 여건에선 **해상도가 디테일의 직접 레버**다.
- **24는 배제:** `TILE=32`에서 32÷24=1.33 → 정수배 불가 → 픽셀 시머. 아키텍처(맵 좌표·충돌·전 레이아웃·프레이밍)를 통째로 바꿔야 함. **`TILE=32` 하에서 깔끔한 선택은 16 또는 32뿐** — 디테일 우선이라 32.

즉 [ADR-0049]가 민 "청키 그레인"의 전제(손그림)가 우리(AI)에겐 성립하지 않아, 16px는 스타듀에 **가까워지기는커녕 멀어졌다**.

## 결정

**환경/전 인게임 아트의 논리 해상도를 32px native로 되돌린다**([ADR-0013] 규격). `TILE=32`·아트 파일 32px 1:1 드로우·뷰포트 960×540 ×2 스트레치(온스크린 64px)는 전부 불변 — **바뀌는 것은 아트의 유효 해상도(16→32)뿐**.

- **청키화 폐기:** `chunkify_asset.py`(÷2 BOX→×2 NEAREST)의 16 밀도 강제를 중단한다. 객체는 32px native로 둔다(스타듀 손그림 청키함은 우리에 부적용 — AI는 해상도로 디테일을 낸다).
- **신규 아트 처리 규칙(부드러움 우선):** AI 생성본을 32-native로 처리할 때 **과한 양자화·하드 외곽선 패스 금지**. 가벼운 정리(≈24색 median + 알파 하드에지)로 계조·부드러움을 보존한다(스타듀 룩). debris 파이프라인(`tools/_debris16_build_v6.py`)이 참조 구현.
- **1칸 규격 유지:** 1칸 프롭 = 32×32 파일(TILE=32에 1:1). 논리=32라 파일=32에 네이티브(추가 스케일 없음).

## 결과 (downstream)

- **표준 = 32-native(전 에셋).** 32-native는 [ADR-0049] 실험 前의 상태([ADR-0013])라 **대부분 에셋은 이미 32-native**(git). [ADR-0049] 16px는 **안식 농원 한 슬라이스만 롤아웃**됐으므로(PR #195), 32-native 통일 = **그 슬라이스의 16px 변경만 표준에 맞추기**:
  - ①`chunkify_asset.py`로 청키화한 오브젝트 → **git 32-native 원본 복원**(간단).
  - ②16px 소프트 지면(`terrain16/`·`_build_ground16`) → **32-native 지면으로**.
  - ③나머지 전부 + 신규 아트(debris 등) = **32-native 그대로**.
- 즉 "전 에셋 대공사"가 아니라 **표준 확정 + 어긋난 16px 슬라이스 정합**. onscreen 64px·프레이밍·`TILE=32`·좌표·충돌·세이브 불변.
- **debris 9종(3×3):** 1칸 32-native로 확정 — 잡초(꽉 찬 잎 로제트)·업화석(볼더+3/4조각 무더기)·석화 고목(대각선 통나무). PixelLab 생성 + 32-native 처리. 배선(2×2→1×1 충돌·좌표·발치바) 후행.
- **[ADR-0013] tileset-ruleset §1** 재유효화(32 native). **[ADR-0049]는 이 ADR로 대체.**
- **onscreen 64px·프레이밍(~30타일 폭)·`TILE=32`·좌표·충돌·세이브 불변** — [ADR-0049]와 동일하게 코어 렌더는 안 건드린다.

## 감수하는 트레이드오프

- 오늘 한 16px 실험(안식 농원 한 슬라이스, PR #195)을 **32-native 표준에 맞춘다**(하루 안의 방향 전환). 대부분 에셋은 이미 32-native라 실작업은 그 슬라이스에 한정. 대신 owner가 원하는 **스타듀식 디테일·부드러움**을 얻는다("끝까지 플레이" 원칙엔 무영향 — 순수 룩 결정).
- "청키 그레인" 미학은 포기. 단 실제 스타듀도 극단적 청키가 아니며(손그림 16px는 깨끗함), 우리 북극성은 스타듀 스크린샷이므로 32-native가 정합.
