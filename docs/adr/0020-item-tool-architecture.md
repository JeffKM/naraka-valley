# 아이템·도구 아키텍처: 데이터 주도 아이템 시스템 + 선택 도구 모델, 스튜디오 안 만듦

Phase 3 전 UI 작업(핫바·인벤토리) grill(2026-06-24)에서, 그 모든 게 깔고 앉은 *기반*이 아직 없음이 드러났다 — 현재 `inventory.gd`는 작물 수확물·씨앗 두 Dictionary뿐이고, "도구"는 `field.next_action()`이 칸 상태를 보고 자동 분기하는 **단일 컨텍스트 키 `E`** 흐름에 암묵적으로 박혀 있을 뿐 실체가 없다(M2.3도 "아이템 시스템 생기면 추가"라고 seam만 기록). 핫바(12칸 선택 슬롯)·Phase 3 루프(낚시·채광 도구)·콘텐츠 저작이 전부 이 기반을 전제한다.

## 결정

**① 데이터 주도 아이템 시스템을 세운다.** `crops.gd`(CropCatalog)를 일반화한 `ItemCatalog`(static 데이터 또는 `.tres` Resource) — id·표시명·아이콘·카테고리·스택여부·가격을 데이터로. 카테고리: **도구**(스택 불가·유니크·핫바 점유) / **씨앗** / **수확물·산물** / **재료**(광물·목재·채집물, Phase 3) / **소모품**(음식). `quality` 필드는 *예약만*(기본=일반, 등급 구현은 Phase 3 §2.1 — [ADR-0019] 결). 아이템은 "다 만들기"가 아니라 *루프가 요구할 때 한 종류씩* 추가(그레이박스 먼저).

**② 단일 컨텍스트 `E` → 스타듀식 "선택 도구" 모델로 완전 피벗.** 핫바에서 선택한 도구가 곧 "손에 든 것"이고, 그 도구에 맞는 동작만 앞 칸에 작용한다(괭이 선택 시 괭이질만). 농사 입력·온보딩·`field`의 `next_action` 강제 순서 회귀를 재작성한다.

**③ 별도 콘텐츠 "스튜디오"를 만들지 않는다([ADR-0001](./0001-pixel-art-ai-pipeline.md) 재확인).** 맵 배치 = Godot의 TileMap·씬 에디터, 아이템 정의 = 인스펙터에서 `.tres` 편집 — *Godot 에디터가 이미 그 스튜디오다.* 정말 인게임 배치가 필요하면 얇은 디버그 스폰 모드 한 겹만 허용(글루 스크립트 범위).

## Considered Options

- **(B) 하이브리드(기각):** 핫바엔 도구를 *표시*만 하고 작용은 여전히 `next_action`이 자동 분기. 조작감·회귀 보존이 장점이나, "도구 선택"이 장식이 되고 낚시·채광 도입 시 농사 입력을 *다시* 갈아엎어야 한다 — 피벗을 미루는 비용만 키운다. 그래서 지금 (A) 완전 피벗.
- **별도 스튜디오 앱(기각):** Godot 에디터가 공짜로 주는 맵 배치·Resource 편집을 재발명. ADR-0001의 야크 셰이빙 금지 정신에 반함. 비개발자 협업 같은 *Godot 에디터로 안 되는* 구체 필요가 확인되면 그때 재검토.

## Consequences

- **`field.next_action` 강제 순서 설계가 폐기된다.** T2.1~T2.3·온보딩(T4.1) 회귀가 선택 도구 모델로 재작성된다 — Phase 2.8의 핵심 리스크.
- **핫바·인벤토리 UI가 이 아이템 시스템 위에서만 의미를 갖는다.** UI 코어(§3.4 컨텍스트 스위칭: 하단 공통 가방 고정 + 상단 레이어 교체)는 이 시스템을 데이터 원천으로 삼는다.
- **아이템 아이콘 = 에셋.** 어떤 아이템이 있는지는 이 시스템이 정의하고, 그 아이콘 도트화는 에셋 패스(Phase 2.8 말미 또는 Phase 2.7 결)로 따라온다.
- **이 모델을 단일키 컨텍스트로 되돌리거나 자체 스튜디오를 만들려면** 이 ADR을 새 ADR로 대체해야 한다.
